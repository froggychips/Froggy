# Design: Menubar объяснимости

| Поле | Значение |
|---|---|
| Статус | Draft |
| Фаза | Уровень 1.5 — Trust Governance |
| Зависит от | [`activity-detection.md`](activity-detection.md), [`freeze-confidence-policy.md`](freeze-confidence-policy.md) |
| Связано с | [`THESIS.md`](../THESIS.md) |

## Зачем это нужно

Доверительный слой не существует, если пользователь его не видит.

`ActivityDetector` и `FreezePolicyEngine` производят структурированные
решения с богатыми трейсами. С точки зрения пользователя, всё это
не имеет значения, пока он не может ответить на два вопроса менее чем
за пять секунд:

- **«Что Froggy сейчас делает с моим Mac?»**
- **«Почему Slack исчез / ведёт себя странно?»**

Если на эти вопросы нет честных, немедленных, понятных человеку ответов —
Froggy по определению из [`THESIS.md`](../THESIS.md) психологически
враждебен, независимо от того, насколько хороша лежащая в основе логика
принятия решений.

Этот документ охватывает **слой представления**. В нём нет никакой
бизнес-логики. Принятие решений происходит выше; этот слой лишь
*показывает*, что произошло и почему.

## Цели

1. Отображать каждое решение о заморозке в реальном времени с *реальной
   причиной* (взятой из `DecisionTrace`), а не шаблонным приближением.
2. Показывать статус первым; предлагать детализацию любопытным.
3. Делать каждое показанное число и временную метку *прослеживаемыми*
   до конкретного сигнала — никакого придуманного контекста, никакой
   округлённой информации, которую нельзя восстановить.
4. Быть считываемым с первого взгляда. Пользователь не должен читать абзацы,
   чтобы понять текущее состояние. Заголовки — вперёд, детали — по запросу.
5. Поддерживать локализацию. Сначала английский, структура, позволяющая
   добавить русский перевод без переархитектуры.

## Вне scope

- **Не панель управления.** Отдельная поверхность управления заморозкой/размораживанием
  вне контекста объяснения — за рамками (существующий menubar уже имеет Thaw All).
  Однако **контекстные действия per-строке, напрямую привязанные к объяснению**,
  *в scope* — см. L3 ниже. Правило: встроенное действие допустимо, если его
  смысл однозначен при наличии объяснения рядом («разморозить именно этот Slack,
  о котором ты мне только что рассказал»). Всё более абстрактное — в другое место.
- **Не дашборд метрик.** Датчики pressure и суммарная освобождённая RAM —
  полезный контекст, но задача Froggy — не заменять Activity Monitor.
- **Не генератор спама уведомлений.** Системные уведомления зарезервированы
  для событий, о которых пользователю нужно знать *прямо сейчас*
  (см. «Правила уведомлений» ниже).
- **Не просмотрщик логов.** Полные структурированные логи идут в `os_log` /
  Console; menubar показывает кураторское, человекочитаемое подмножество.

## Информационная архитектура

Четыре уровня с прогрессивной детализацией:

| Уровень | Где | Содержание | Когда пользователь видит |
|---|---|---|---|
| **L1: Взгляд** | Иконка MenuBar + tooltip | Состояние лягушки (idle / managing / critical), количество замороженных | Всегда видно |
| **L2: Статус** | Верхняя часть выпадающей панели | «3 приложения заморожены, 1.2 ГБ освобождено, pressure: warning (4 мин)» | Клик на иконку menubar |
| **L3: Детали** | Per-app строка в выпадашке | «Slack — заморожен 18 мин назад, ~600 МБ освобождено, скоро попытка разморозки» | Наведение или клик на приложение |
| **L4: Трейс** | Раскрытый вид per-app | Полный `DecisionTrace`: оценённые сигналы, пороги, состояние бюджета | Клик «почему?» в L3 |

Пользователь уделяет внимание пропорционально контексту. L1 отвечает
«делает ли Froggy что-то странное» за 0.5 с. L2 отвечает «что происходит»
за 3 с. L3 отвечает «что с *этим* конкретным приложением» за 5 с.
L4 отвечает «объясни детально» для любопытных или для баг-репортов.

## L1: Состояние «с первого взгляда»

Иконка лягушки адаптируется к текущему состоянию Froggy:

| Состояние | Иконка | Tooltip |
|---|---|---|
| Idle (модель не загружена) | 🐸 (по умолчанию) | «Froggy idle» |
| Active (модель загружена, нет заморозок) | 🐸 (лёгкий пульс при генерации) | «Модель загружена, нет замороженных приложений» |
| Managing (≥1 заморожено, pressure нормальный/warning) | 🐸 (вариант иконки) | «2 приложения заморожены — управление pressure» |
| Critical (pressure критический, заморозки активны) | 🐸 (вариант + акцентный цвет) | «3 приложения заморожены — память критическая» |
| Anomaly (заморозка не удалась, ошибка decision-engine) | 🐸 (значок предупреждения) | «Проблема с последним решением о заморозке» |

Текстовые варианты (без пролиферации эмодзи) — максимум три комбинации
SF Symbol. Эмодзи 🐸 — строго иконка приложения, не индикатор состояния.

## L2: Заголовок статуса

Одна строка, всегда в верхней части выпадашки:

```
[Сводка состояния]  ·  [Pressure]  ·  [Освобождено]
```

Конкретно:

```
3 apps frozen  ·  warning (4 min)  ·  ~1.2 GB recovered
```

В режиме idle:

```
No apps frozen  ·  pressure: normal  ·  Froggy ready
```

При critical:

```
3 apps frozen  ·  CRITICAL (just now)  ·  ~1.2 GB recovered  ·  considering MLX unload
```

Правила:

- «Освобождено» — **оценка**, не точное измерение. Использовать префикс `~`
  для обозначения приближения.
- Время pressure — продолжительность текущего уровня pressure: «warning (4 мин)»,
  не «warning с 14:23».
- Опциональный 4-й сегмент («considering MLX unload») появляется только
  в конкретных переходных состояниях. Никогда спекулятивно — только если
  policy engine действительно перешёл к этому следующему действию.

## L3: Строка per-app

Для каждого приложения, текущего замороженного или недавно рассмотренного:

```
┌─────────────────────────────────────────────────────────┐
│ Slack                       [why?] [thaw] [never freeze]│
│ Frozen 18 min ago · ~600 MB freed · thaw in ~4 min      │
└─────────────────────────────────────────────────────────┘
```

Для приложения, *рассмотренного но пропущенного*:

```
┌─────────────────────────────────────────────────────────┐
│ Spotify                                        [why?]   │
│ Skipped 2 sec ago · keeping active                      │
└─────────────────────────────────────────────────────────┘
```

Строки пропущенных приложений видны лишь недолго (~30 с) — они отвечают
на вопрос «я только что почувствовал что мой Mac притормозил, что изменилось?»,
но не засоряют длительный вид.

### Встроенные действия для замороженных строк

Два действия per-строке рядом со ссылкой `[why?]`, только для строк
текущих замороженных приложений:

- **`[thaw]`** — немедленный `SIGCONT` для *этого конкретного приложения*,
  минуя timing автоматической разморозки на основе pressure. В существующем IPC
  есть `thawAll`; здесь нужна новая команда (см. «Добавления API» ниже).
  Действие **не** добавляет приложение ни в какой список исключений — это
  одноразовое переопределение для данной заморозки, при следующем событии
  pressure приложение будет рассмотрено снова.
- **`[never freeze]`** — добавляет `bundleId` приложения в `freezeExclusion`
  в `FroggyConfig` (определено в
  [`freeze-confidence-policy.md`](freeze-confidence-policy.md)) и
  инициирует немедленную разморозку. После этого policy engine откажется
  рассматривать это приложение вообще. Toast подтверждения: «Slack добавлен
  в список исключений из заморозки».

Эти действия привязаны к контексту объяснения — пользователь видит *почему*
приложение заморожено, и естественным продолжением является «на самом деле,
не делай этого». Предоставление управления где-либо ещё — вне scope.

Третье потенциальное действие — **`[lower threshold]`**, которое записало бы
в `activityConfidenceOverride` для снижения агрессивности без полного исключения —
отложено. Его сложнее объяснить в одной метке кнопки, а два более грубых
действия покрывают реалистичные сценарии использования.

Пропущенные строки не имеют встроенных действий: пользователь уже получил
желаемый результат (приложение продолжает работать), никаких дальнейших
действий не требуется.

## L4: Трейс per-решение

Когда пользователь нажимает «why?» — раскрываем встроенно (не модально —
модальное окно нарушает поток). Рендерим `DecisionTrace` из
[`freeze-confidence-policy.md`](freeze-confidence-policy.md):

```
Slack — frozen 18 min ago

  Decision: freeze
  Pressure level: warning
  Tier: 1
  Threshold to skip: 0.30
  Confidence score: 0.12   (below threshold → eligible)

  Signals contributing to confidence:
    • frontmost            → no               (weight 1.0,  contrib 0.0)
    • audio-active         → no               (weight 0.9,  contrib 0.0)
    • camera-active        → no               (weight 0.95, contrib 0.0)
    • recent-input         → 87 sec ago       (weight 0.7,  contrib 0.05)
    • recent-frontmost     → 18 min ago       (weight 0.4,  contrib 0.0)
    • network-active       → 2 sockets idle   (weight 0.3,  contrib 0.07)
    • cpu-burst            → 0.3% in last 5s  (weight 0.2,  contrib 0.0)

  Budget check: 4 min used of 5 min/hour → eligible
  Cooldown check: 12 min since last freeze → eligible
  Override check: none

  Action taken: SIGSTOP + jetsam pageout
  Pageout result: succeeded (~600 MB freed via compressor)
  Will reconsider thaw in: 4 min (or earlier if pressure drops)
```

Рендерится непосредственно из трейс-JSON. **Никакой строковой интерполяции,
вносящей информацию, отсутствующую в трейсе.** Это строгое правило, предотвращающее
расхождение между объяснением и реальностью: если трейс это говорит — отображаем;
если трейс этого не говорит — не придумываем.

## Обновления в реальном времени

Menubar подписывается на `liveDecisions()` AsyncStream от policy engine
(определено в [`freeze-confidence-policy.md`](freeze-confidence-policy.md))
и существующий поток pressure от `MemoryPressureMonitor`.

Стратегия:

- L1 (иконка) обновляется при каждом переходе состояния — с debounce ≥ 200 мс
  во избежание мерцания при быстрых изменениях pressure.
- L2 (заголовок) обновляется при каждом релевантном событии — изменение
  pressure, заморозка, разморозка, изменение оценки освобождённого.
- L3 (строки) анимируются появлением/исчезновением через SwiftUI transitions —
  замороженные приложения появляются с лёгким slide; пропущенные появляются
  ненадолго и затем исчезают.
- L4 (трейс) загружается по запросу; кэшируется в памяти для сессии.

Polling: **нет**. Push-based через стримы. При обрыве стрима
(перезапуск daemon'а) — показывать пилюлю «reconnecting…» в заголовке
на всё это время.

## Правила уведомлений

Скупо и заслуженно. По умолчанию — **тихая работа**: иконка menubar —
основная поверхность. Системные уведомления срабатывают только для:

| Событие | Уведомление | Обоснование |
|---|---|---|
| Pressure эскалировал до critical И заморозки не дали достаточного освобождения | «Froggy: память критическая, освобождаю фоновые приложения» | Пользователь, скорее всего, вот-вот почувствует это; подтверждение успокаивает |
| Бюджет заморозки исчерпан для ≥1 приложения при сохраняющемся pressure | «Froggy: не могу освободить больше памяти без вмешательства в активные приложения» | Это редкое, но реально требующее действий состояние — пользователь, возможно, захочет вручную закрыть что-то |
| Заморозка не удалась (например, ошибка pageout) | «Froggy: не удалось заморозить [App] — см. menubar» | Отладочный сигнал |
| Сработал activity-canary (мы не должны были заморозить это приложение) | «Froggy: разморозил [App], обнаружена аудиоактивность» | Честное признание бага выше по стеку; сохраняет доверие |

Без уведомлений для:

- Штатных заморозок / разморозок (стандартный поток). Они появляются в menubar,
  но не прерывают работу.
- Переходов pressure между normal и warning. Слишком частые, чтобы быть
  значимыми.

## Генерируемый текст — язык и тон

Три правила:

1. **Конкретно, не расплывчато.** «Slack frozen — memory critical (6.8/8 GB
   used), no active call detected, background 18 min» лучше, чем «Slack
   has been suspended due to high memory usage».
2. **Честность в отношении неопределённости.** «Spotify kept active — audio session
   open» — честно. «Spotify protected from freeze» — маркетинг.
3. **Никакого жаргона на пользовательском уровне.** L1–L3 говорят «memory critical»,
   не «MEMORYSTATUS_PRESSURE_CRITICAL». L4 (трейс) может использовать технические
   имена, потому что L4 предназначен для пользователей, отлаживающих поведение.

Шаблоны живут в одном Swift-файле (`ExplanationFormatter.swift`),
ключи — enum-кейсы `DecisionTrace`. Каждый вариант шаблона имеет
тестовый fixture: при трейсе `X` генерируемый текст — `Y`. Snapshot-тесты
держат это честным.

## Локализация

Фаза 1 (в рамках этого дизайна): только английский. Шаблоны в
`Localizable.strings` с первого дня — без хардкода литералов — чтобы
фаза 2 была чистым переводом, не рефакторингом.

Кандидат на фазу 2: русский. В кодовой базе уже есть двуязычные конвенции
(разделённый README, комментарии на русском). Один дополнительный файл
`Localizable.strings(ru)`, когда появится желание его поддерживать.

Другие языки: отложены до появления спроса от внешних контрибьюторов.

## Добавления API

### IPC

Команды только для чтения для поверхности объяснения:

```
decisions [--limit N]
  Список последних решений, новейшие первые. Вывод: JSON-массив
  DecisionTrace.

decision <id>
  Одно решение по id. Вывод: JSON DecisionTrace.

decisionsLive
  Стриминг. Пушит новые JSON-строки DecisionTrace по мере появления
  решений. Используется menubar; может использоваться внешними инструментами.
```

Команды с изменением состояния для встроенных действий L3:

```
thaw <pid>
  Немедленный SIGCONT для одного pid. Отличается от существующего
  thawAll. Проверяет, что pid в данный момент заморожен Froggy
  (отказывает на неизвестный pid, предотвращая эскалацию через эту команду).

addExclusion <bundleId>
  Добавляет bundleId в FroggyConfig.freezeExclusion, сохраняет в
  config.json и инициирует немедленную разморозку, если приложение
  сейчас заморожено. Идемпотентен. Используется действием [never freeze].

removeExclusion <bundleId>
  Обратное к addExclusion. По умолчанию не экспонируется в menubar, но
  нужен, чтобы конфиг можно было редактировать людьми через ту же IPC-поверхность
  (избегает необходимости ручного редактирования JSON).
```

Endpoint decisions также публично полезен: баг-репорты становятся проще,
когда пользователь может приложить `froggy decisions --limit 50 > log.json`,
не раскрывая больше, чем намеревается.

Эндпоинты с изменением состояния соблюдают существующую модель доверия IPC —
Unix-сокет с файловыми правами, без аутентификации; та же граница доверия,
что и у остального daemon'а.

### Вью `FroggyMenuBar`

```swift
struct FreezeStatusHeaderView: View      // L2
struct FrozenAppsListView: View          // L3
struct DecisionTraceView: View           // L4
struct LivePressureGauge: View           // правый сегмент L2
```

Каждое независимо предпросматриваемо в SwiftUI Preview с fixture-данными —
fixtures живут в `Tests/FroggyMenuBarTests/Fixtures/`.

## Режимы отказа

| Отказ | Обнаружение | Поведение |
|---|---|---|
| Стрим `decisionsLive` обрывается | Reader ловит конец стрима | Пилюля «Reconnecting…» в заголовке; авто-повтор каждые 2 с |
| В решении отсутствуют поля (дрейф схемы) | Частичный JSON decode | Показываем fallback с сырыми полями в L4; ухудшенный, но видимый вид |
| Генератор L4 не может отрендерить трейс (неизвестный enum case) | Исчерпывающий switch | Показываем сырой JSON как последнее средство + телеметрия |
| Daemon offline | Первоначальное соединение не удалось | Статическое состояние «Daemon not running» в menubar; предложить «Start daemon», если PIDFile присутствует |

Menubar **никогда не должен крашиться из-за плохих данных daemon'а**. Он может
показывать ухудшенные вью, откатываться к сырому JSON, отказываться рендерить —
но не крашиться.

## Plan тестирования

Snapshot-тесты:

- Для каждого enum-кейса `DecisionTrace` — fixture `.json` + ожидаемый
  отрендеренный текст. Тесты утверждают побитовую стабильность генерации.
  Намеренные изменения копии явно регенерируют fixtures.
- Покрытие: минимум один fixture per case в `FreezeReason`,
  `SkipReason`, `ThawReason`. Оба языка — английский и (при добавлении) русский.

UI-тесты:

- Smoke-тесты на основе SwiftUI Preview через пакет `swift-snapshot-testing`.
  Каждое вью рендерится на fixtures, image diff в CI. Опционально — только
  если стоимость поддержки приемлема.

Доступность:

- У каждого динамического значения есть VoiceOver-метка с объяснением содержимого.
  Трейс L4 экспонирует сигналы как правильно структурированный список, а не
  плоский текстовый дамп.
- Цвет никогда не является единственным носителем состояния (варианты иконки
  несут реальную нагрузку, акцентный цвет — дополнительный).

Ручное тестирование:

- Тест на реальном устройстве: принудительно вызываем pressure (`memory_pressure -l warn`),
  проверяем корректность menubar; снимаем pressure, наблюдаем поток разморозки.
  Пишем одностраничный «скрипт ручного тестирования», чтобы это было воспроизводимо.

## Поэтапная реализация

| ID | Scope | Приёмка |
|---|---|---|
| EXP-1 | IPC-команды `decisions` + `decisionsLive`; in-memory ring buffer (последние 100) | `froggy decisions --limit 5` показывает реальные последние решения; live stream эмитирует при новых |
| EXP-2 | L1 + L2 в MenuBar (состояния иконки, заголовок статуса) | Взгляд + статус обновляются вживую при изменении pressure / заморозок |
| EXP-3 | L3 (строки per-app) | Замороженные приложения + недавно пропущенные рендерятся со сводками |
| EXP-4 | L4 (раскрытие трейса) | Ссылка «why?» раскрывает трейс встроенно с полными вкладами сигналов |
| EXP-5 | Правила уведомлений (4 события) | Critical pressure / исчерпан бюджет / решение не удалось / canary сработал — все появляются как уведомления |
| EXP-6 | Извлечение `Localizable.strings` + ревью англоязычного текста | Весь пользовательский текст проходит через локализацию, английская копия проверена на тон |

EXP-1 + EXP-2 + EXP-3 — минимально жизнеспособный trust UX. EXP-4 —
уровень «опытного пользователя / баг-репорта». EXP-5 охватывает редкие, но
важные события. EXP-6 — структурная очистка, которую следует сделать не позднее
EXP-3, чтобы избежать ретрофита.

## Открытые вопросы

1. **Сохранять ли историю решений между перезапусками?** Сейчас ring buffer
   только в памяти. Плюсы persistence: лучшие баг-репорты, ответ на «что было
   ночью». Минусы: площадь приватности (решения ссылаются на bundle id
   и временные метки). Склоняемся к: только в памяти по умолчанию, флаг
   opt-in persistence в конфиге.
2. **Использовать ли SF Symbols или оставить эмодзи лягушки для L1-иконки?**
   Эмодзи лягушки — бренд. SF Symbols — нативный macOS и нагляднее.
   Возможный компромисс: эмодзи лягушки как база, overlay SF Symbol бейджа
   для состояния. Стоит взгляда дизайнера.
3. **Место ли L4 в menubar вообще?** Он плотный; мог бы жить в отдельном окне,
   вызываемом через «Open trace inspector». Trade-off: встроенное держит всё
   в одном месте; отдельное окно позволяет более богатый рендеринг. Склоняемся
   к встроенному пока (одно место для просмотра), пересмотр после dogfood.
4. **Что насчёт «длинного нарратива» о «что происходило последний час»?**
   Потенциальный L5 — timeline-вью с резюме «Slack заморожен 3 раза, всего
   22 мин, освобождено ~1.8 ГБ» и т.д. Полезно, но риск scope creep.
   Откладывается до появления реального L1–L4.

## Связь с THESIS

Согласно [`THESIS.md`](../THESIS.md), доверительный слой *сам по себе*
является первой пользовательской возможностью Уровня 1.5. Этот документ —
механизм, с помощью которого «trust governance» становится чем-то, что
пользователь *видит*, а не только тем, что daemon *делает*.

Конкретно:

- Это **качественное**: ни один другой инструмент на macOS не показывает
  per-app трейс *почему* процесс был ограничен, со структурированными
  вкладами сигналов. Activity Monitor показывает *что*; Console показывает
  *фрагменты*; это показывает *почему*.
- Это **доказательство** того, что обнаружение активности + policy стоило
  строить. Без объяснимости эти два слоя — чёрный ящик, и у пользователя
  нет причин им доверять.
- Это явно противостоит «ловушке инфраструктурной гравитации» из
  [`THESIS.md`](../THESIS.md): menubar — это **выпущенная, пользовательская
  функция**. В день когда EXP-3 приземлится, Уровень 1.5 произведёт
  что-то, чем автор пользуется каждый день и что другие люди могут
  немедленно понять.

Тройка `(обнаружение активности → policy → объяснимость)` замыкается здесь.
После этого субстрат произвёл свою первую возможность end-to-end. Следующее
решение — не «больше субстрата», а «какую качественную возможность поверх
этого мы выпускаем первой».
