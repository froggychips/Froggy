# Design: Обнаружение активности для Freeze Confidence

| Поле | Значение |
|---|---|
| Статус | Draft |
| Фаза | Уровень 1.5 — Trust Governance |
| Зависит от | Mem-1 (`MemoryPressureMonitor`), Mem-2 (`PageoutChain`) — оба смёржены |
| Связано с | [`THESIS.md`](../THESIS.md), предстоящий `freeze-confidence-model.md`, `explainability-menubar.md` |

## Зачем это нужно

Согласно [`THESIS.md`](../THESIS.md), заморозка без доверия — психологически
враждебная система. Единственный конкретный сценарий отказа, способный
уничтожить проект: **Froggy замораживает Slack во время звонка, аудио Zoom
обрывается, пользователь удаляет приложение и рассказывает всем, что Froggy —
«та лягушка, которая сломала мне встречу»**. Одного такого инцидента достаточно.

Митигация — не «замораживать меньше»: это обрушивает тезис. Митигация:
**не замораживать процесс, который делает что-то важное для пользователя
прямо сейчас**. Обнаружение активности — входной слой, делающий это возможным.

Этот документ охватывает только сбор сигналов и вычисление confidence.
Логика *принятия решений* (как Vortex использует confidence для управления
попытками заморозки) и логика *объяснения* (как menubar представляет
произошедшее и причины) — отдельные документы.

## Цели

1. Вычислять **confidence score активности per-PID** в диапазоне `[0.0, 1.0]`,
   где выше — значит «пользователь активно занят этим процессом прямо сейчас».
2. Семплировать с низкими затратами — раз в ~2 с без ощутимого влияния
   на RAM, CPU и батарею на M1/M3 Air.
3. Быть **наблюдаемым**: каждое решение о заморозке должно быть прослеживаемо
   до конкретных сигналов, сформировавших его confidence score.
4. Деградировать gracefully при недоступности любого источника сигнала
   (Apple убрала приватный API, отозвано разрешение AX и т.д.) — откатываться
   к оставшимся сигналам, никогда не блокировать пайплайн.
5. Всё вычисление держать **локально**. Никакие данные об активности
   не покидают машину.

## Вне scope

- **Не** универсальный монитор активности процессов. Мы скорим только
  кандидатов, которых пайплайн заморозки собирается рассмотреть.
- **Не** обучающаяся система. Никакого ML, никакого per-user обучения.
  Rules-based взвешенный scorer — достаточен и объясним.
- **Не** замена `MemoryPressureMonitor`. Pressure решает *нужна ли* заморозка;
  активность решает *какие процессы подходят*.
- **Не** абсолютная точность. Ложноположительные срабатывания (отказ
  замораживать что-то реально простаивающее) допустимы. Ложноотрицательные
  (заморозка того, чем занят пользователь) — *нет*.

## Место в стеке

```
┌─────────────────────────────────────────────────────────────┐
│ MemoryPressureMonitor   →   сигнал "warning" / "critical"   │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ VortexCoordinator.applyPolicy(level)                        │
│   for each candidate pid in tier-N:                         │
│     score = ActivityDetector.confidence(pid)  ◀── этот doc  │
│     if score >= tier-threshold: skip freeze                 │
│     else: vortex.freeze(pid, reason: explanation)           │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ Vortex.freeze(pid)  →  PageoutChain  →  FrozenPidsStore     │
└─────────────────────────────────────────────────────────────┘
```

`ActivityDetector` — новый актор в `VortexCore`, параллельный
`MemoryPressureMonitor`. Coordinator синхронно запрашивает его per-кандидату
в момент принятия решения о заморозке (дёшево, цель < 5 мс).

## Сигналы

Каждый сигнал возвращает нормализованный вклад в `[0.0, 1.0]`. Итоговый
confidence — **взвешенная сумма, не произведение**: несколько слабых сигналов
должны складываться в сильное вето, и одного сильного сигнала самого по себе
достаточно.

| ID | Сигнал | Источник | API | Вес | Примечания |
|----|--------|----------|-----|-----|------------|
| `frontmost` | Процесс является frontmost-приложением | `NSWorkspace` | public | 1.0 | Жёсткое вето — frontmost **никогда** не замораживается, независимо от других сигналов. |
| `audio-active` | Процесс владеет активным аудио-потоком I/O (ввод микрофона или вывод) | CoreAudio HAL | public | 0.9 | Охватывает Zoom, Teams, FaceTime, Discord voice, Music, Spotify. |
| `camera-active` | Процесс владеет активным потоком камеры | CoreMediaIO HAL | public | 0.95 | Сильнее аудио, потому что видеозвонки — более критичный сценарий. |
| `recent-input` | Время с последнего AX-наблюдаемого пользовательского ввода в окнах этого приложения | AX API | public, требует разрешения Accessibility | 0.7 (затухание за 60 с) | Клавиатурная/мышиная активность per-app. |
| `media-playing` | Процесс является системным «now playing» клиентом | MediaRemote (private) | private | 0.6 | Spotify, Music, вкладки браузера с HTML5 audio. |
| `network-active` | Высокое число established TCP-сокетов + недавний трафик | `proc_pidinfo` | public | 0.3 | Эвристика. Не перевешивать: у Slack всегда открытые сокеты. |
| `fullscreen` | Процесс владеет окном текущего fullscreen-пространства | Quartz Window Services | public | 0.5 | Не замораживать браузер, демонстрирующий слайды. |
| `recent-frontmost` | Был frontmost последние N секунд | Внутреннее отслеживание | — | 0.4 (затухание за 30 с) | «Пользователь только что переключился» не должно немедленно вызывать заморозку. |
| `cpu-burst` | Процесс использовал > X% CPU за последние 5 с | `proc_pidinfo` (rusage) | public | 0.2 | Слабый сигнал — многое крутится вхолостую. |

### Калибровка весов

Приведённые веса — начальные значения. Они намеренно
**асимметричны**: сигналы, сильно коррелирующие с «пользователь занят»,
получают веса близкие к вето (audio/video); сигналы со слабой корреляцией
(CPU, сеть) — tie-breakers.

Формула агрегированного confidence:

```
confidence = min(1.0, sum(signal_value * signal_weight))
```

Два проектных решения, которые могут удивить:

1. **`frontmost` реализован как жёсткая pre-check, а не взвешенный сигнал.**
   Если кандидат является frontmost-процессом — немедленно вернуть `1.0`,
   пропустить всё остальное. Это нужно и для корректности
   (любая заморозка frontmost — баг), и для скорости (не надо семплировать
   другие сигналы).

2. **Никакого мультипликативного затухания.** Процесс с активным `audio-active`
   должен fail-safe переходить к «не замораживать», даже если все остальные
   сигналы говорят «простаивает». Это принцип асимметричного отказа: ложные
   позитивы дёшевы, ложные негативы — катастрофичны.

## Интеграция confidence с freeze policy

Пороги per-tier (начальные значения, настраиваются через конфиг):

| Уровень pressure | Tier | Порог confidence для пропуска заморозки |
|---|---|---|
| `warning` | tier-1 | `>= 0.3` |
| `critical` | tier-1 | `>= 0.5` |
| `critical` | tier-2 | `>= 0.4` |

Читать так: **при warning даже умеренно активное приложение пропускается.
При critical пропускаются только сильно активные приложения.** Это соответствует
принципу: critical pressure — настоящая аварийная ситуация, где некоторые
потери UX допустимы; warning pressure должен быть практически незаметен
для пользователя.

Отдельное поле конфига `activityConfidenceOverride: [bundleId: Float]`
позволяет пользователям вручную фиксировать конкретные приложения на
более высоких порогах — например, «никогда не замораживать 1Password независимо
от confidence». (Обратите внимание: это отличается от исключения через
`freezeBundleIds` — то не даёт кандидату попасть в пайплайн заморозки вообще;
здесь — переопределение confidence.)

## API

```swift
public actor ActivityDetector {
    public init(
        signalSources: [ActivitySignalSource] = .defaults,
        clock: any Clock<Duration> = ContinuousClock()
    )

    /// Возвращает confidence score и структурированный трейс о том, как он
    /// был вычислен. Трейс — входные данные для слоя объяснимости.
    public func confidence(forPid pid: pid_t) async -> ActivityConfidence
}

public struct ActivityConfidence: Sendable {
    public let pid: pid_t
    public let score: Float                  // 0.0 ... 1.0
    public let signals: [SignalContribution] // заполняется для объяснимости
    public let sampledAt: Date
}

public struct SignalContribution: Sendable {
    public let id: String          // "audio-active", "frontmost" и т.д.
    public let value: Float        // сырое значение сигнала, 0.0 ... 1.0
    public let weight: Float       // применённый вес
    public let contribution: Float // value * weight, вклад в сумму
}

public protocol ActivitySignalSource: Sendable {
    var id: String { get }
    var weight: Float { get }
    func sample(forPid pid: pid_t) async throws -> Float
}
```

Каждый источник сигнала — отдельный тип, реализующий `ActivitySignalSource`.
Форма намеренно совпадает с `PageoutImpl` / `MemoryPressureSource` с точки
зрения тестируемости — фейки тривиально подставляются в xctest.

## Стоимость семплирования и параллелизм

Цель: **`confidence(forPid:)` возвращает результат менее чем за 5 мс в типичном случае**.

Стратегия:

- Параллельный запуск сигналов через `withTaskGroup`. Суммарное wall-clock время —
  `max(signals)`, а не сумма.
- Каждый источник сигнала поддерживает собственный внутренний кэш там, где данные
  системные (например, frontmost-приложение меняется не чаще раза в секунду;
  состояние audio HAL меняется редко). TTL кэша — 500 мс.
- Per-PID запросы (CPU burst, сеть, AX input) не кэшируются — они дёшевы,
  и устаревшие данные здесь опасны.

Если какой-либо источник сигнала превышает таймаут 50 мс — возвращает 0
(нейтральное значение) и логирует предупреждение. Пайплайн никогда не блокируется
из-за медленного сигнала.

## Режимы отказа

| Отказ | Обнаружение | Fallback |
|---|---|---|
| AX permission отозвано во время работы | Первый семпл возвращает ошибку | Сигнал `recent-input` возвращает 0, однократный лог, работа продолжается |
| Приватный API MediaRemote сломан в будущей macOS | Поиск символа не удался | Сигнал `media-playing` возвращает 0, однократный лог, работа продолжается |
| Запрос к Audio HAL превысил таймаут | Таймаут 50 мс | Сигнал возвращает 0, повтор на следующем семпле |
| `proc_pidinfo` отказал (sandboxed pid) | Errno EPERM | Сигнал возвращает 0, запрос — best-effort |
| Процесс завершился между выбором кандидата и семплированием | `kill -0` вернул ESRCH | Вернуть `confidence = 0.0` немедленно, пропустить все сигналы |

Режим отказа, который **недопустим**: единственный сломанный сигнал молча
отключает все остальные. Каждый сигнал изолирован независимо и устойчив к отказам.

## Аспекты приватности

Большинство сигналов по своей природе per-PID и не захватывают контент.
Два исключения:

- **`recent-input` через AX API** теоретически может наблюдать, что вводит
  пользователь. Мы семплируем только *временные метки*, не события. AX-наблюдатель
  настроен на получение уведомлений, после чего немедленно записывает `Date()`
  и отбрасывает payload события. Никакие нажатия клавиш, никакие координаты
  кликов никогда не покидают актор.
- **MediaRemote** может возвращать названия треков и URL обложек. Мы намеренно
  **не вызываем** никакие API для получения информации о треке — только
  `MRMediaRemoteGetNowPlayingApplicationPID`. Сигнал означает «в процессе X
  *что-то* играет», но не «играет трек Y».

Оба ограничения проверяемы: юнит-тест утверждает, что
`SignalContribution.id == "recent-input"` никогда не несёт данных помимо
дельты временной метки, а `media-playing` никогда не логирует ничего, кроме PID.

## Что намеренно не обнаруживается (и почему)

- **Глобальное время простоя экрана** (`CGEventSourceSecondsSinceLastEventType`
  глобален, не per-app). Косвенно уже охвачено `recent-input` per-app и
  отслеживанием frontmost.
- **Форма сетевого трафика** (например, «большой объём = активный поток»).
  `proc_pidinfo` даёт количество сокетов дёшево, но per-second байт-счётчики
  дороги. Не стоит того ради маргинального сигнала.
- **Активность уведомлений.** Приложения, генерирующие уведомления, были бы
  неплохим сигналом, но API требует расширения Notification Center
  с отдельными entitlements. Откладывается до тех пор, пока это не понадобится
  по другой причине.

## Plan тестирования

Юнит-тесты:

- **Per-signal:** у каждого `ActivitySignalSource` — фейковый OS-слой.
  Тесты проверяют, что значение отображается в задокументированный диапазон
  `[0.0, 1.0]`, включая крайние случаи (0 на входе, максимум на входе, нет данных).
- **Агрегация:** `ActivityDetector` с набором фейковых источников; проверяем
  взвешенную сумму, short-circuit для frontmost и что один медленный источник
  не блокирует остальные.
- **Затухание:** `recent-input` и `recent-frontmost` корректно затухают
  со временем с использованием инжектированных `Clock`.

Интеграционные тесты:

- **Реальные сигналы на реальных PID:** запускаем дочерний процесс с аудио
  (sox -n -d), проверяем, что `audio-active` переключается на высокий confidence
  для этого PID. Помечены как отключённые по умолчанию через
  `FROGGY_RUN_INTEGRATION_TESTS=1` (аудиоустройства в CI — моки).

End-to-end:

- **Freeze policy с confidence:** при форсированном `.warning` pressure
  через фейковый `MemoryPressureSource` симулируем два кандидата, один
  с высоким confidence (например, frontmost), проверяем, что он пропускается,
  пока другой замораживается.

Планка приёмки: confidence-скоринг корректен на **100% юнит-тестов**,
и **все решения о заморозке в E2E сопровождаются непустым трейсом `signals[]`**,
который их обосновывает.

## Поэтапная реализация

Чтобы избежать PR на 1500 строк, работа разбита на части:

1. **AD-1: Skeleton + frontmost + recent-frontmost.**
   - Актор `ActivityDetector`, протокол, типы.
   - Реализует только два простейших сигнала.
   - Подключён к `VortexCoordinator` с консервативными порогами.
   - Приёмка: заморозка текущего frontmost-приложения теперь невозможна.
     Это одно закрывает наиболее позорный режим отказа.

2. **AD-2: Сигналы audio + camera.**
   - Запросы к CoreAudio HAL и CoreMediaIO HAL.
   - Самые высококонфидентные сигналы для use case обнаружения звонков.
   - Приёмка: симулированный звонок (дочерний процесс sox/avfoundation)
     не может быть заморожен.

3. **AD-3: Recent-input через AX.**
   - Настройка AX-наблюдателя, жизненный цикл (отзыв, запуск/завершение приложений).
   - Поток разрешений в MenuBar — явный: «Froggy нужно разрешение Accessibility,
     чтобы определять, когда вы вводите текст в приложении, которое он может заморозить».
   - Приёмка: ввод текста в приложение на 5 с предотвращает заморозку на следующие 60 с.

4. **AD-4: Оставшиеся эвристики (media-playing, network, fullscreen, cpu-burst).**
   - Каждый — отдельный небольшой PR, если нетривиален.
   - Приёмка: агрегированный confidence на реальной нагрузке (Slack в простое vs
     Slack с WebRTC-звонком) показывает чёткое разделение.

5. **AD-5: Настраиваемые пороги + конфиг `activityConfidenceOverride`.**

Граница фазы с работами по объяснимости: к концу AD-2 трейс confidence уже
заполнен. Menubar с объяснимостью может начать потреблять его независимо,
параллельно.

## Открытые вопросы

1. **Стоимость AX-наблюдения в масштабе.** AX-наблюдатели могут быть дорогими
   при подключении ко многим приложениям. Возможно, нужна ленивая подписка —
   подключаться только к тем кандидатам, которые рассматриваются для заморозки,
   отключаться, когда они перестают быть кандидатами. Будет замерено в AD-3.
2. **Доступ к CoreAudio HAL без root.** Некоторые свойства `kAudioDevice*` могут
   требовать повышенных привилегий для перечисления клиентов. Нужно проверить
   на чистой dev-машине, прежде чем фиксировать `audio-active` как высоковесный сигнал.
   Если заблокировано — откатиться к `lsof` на `/dev/audio*` (работает без root).
3. **Стабильность MediaRemote.** Приватный API; уже под угрозой удаления.
   В AD-4 нужен feature-detect при запуске и graceful degrade при отсутствии
   символов — тот же паттерн, что `memorystatus_control` в Mem-2.
4. **Дефолтные пороги.** 0.3 / 0.5 / 0.4 — первые приблизительные значения.
   Реальные придут из наблюдения за rejection rate заморозок на стабильной
   рабочей нагрузке пользователя в течение недели. Рассмотреть возможность
   экспозиции через `freezeStats` IPC для удобной настройки.

## Связь с THESIS

По качественно-количественному тесту из
[`THESIS.md`](../THESIS.md): обнаружение активности **открывает класс
возможностей**, а не просто улучшает существующий. Без него заморозка —
бинарный риск: либо слишком агрессивная (ломает звонки), либо слишком робкая
(нет преимущества перед Ollama). С ним заморозка становится *управляемой*
операцией, которая масштабируется от мягкой до агрессивной на основе
реальных свидетельств о внимании пользователя. Это фазовый переход, а не
процентное улучшение.

Это также **первая пользовательская возможность** доверительного слоя:
«Froggy не заморозил Slack, потому что у вас активный Zoom-звонок» —
этого не скажет ни один другой инструмент. В сочетании с menubar объяснимости
(отдельный документ) — это то, с чем Froggy выходит в Уровень 1.5.
Это не «инфраструктура до возможностей».
