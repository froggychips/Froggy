# ADR 0004 — Связка Vortex/MLX живёт в Coordinator, а не внутри actor-ов

* **Статус:** Принято (Phase 1)
* **Дата:** 2026-05-06

## Контекст

Главная функция из README («Dynamic RAM Recovery») требует, чтобы перед
загрузкой многогигабайтной MLX-модели Froggy отправлял SIGSTOP фоновым
приложениям, освобождая unified memory. Было два способа реализовать
эту связку:

1. `MLXActor.loadModel(modelPath:)` знает о `VortexActor` и вызывает
   `freezeProcess` со списком pid, полученным откуда-то.
2. `MLXActor` и `VortexActor` ничего не знают друг о друге, а политика
   вынесена в третий actor — `VortexCoordinator`.

## Решение

Вариант 2. `VortexCoordinator` владеет политикой: перечисляет запущенные
приложения по bundle ID (через `NSWorkspace`), замораживает их, затем
ожидает `MLXActor.loadModel`. При сбое размораживает всё, что было
заморожено.

## Последствия

* **Плюс:** `MLXActor` и `VortexActor` остаются независимо тестируемыми.
  `VortexActorTests` не нужна MLX-модель; `MLXActor` (когда появятся
  реальные тесты инференса) не обязан понимать управление процессами.
* **Плюс:** Множество pid, замороженных *для данной загрузки*, отслеживается
  отдельно от pid, замороженных по другим причинам. `emergencyThaw()`
  освобождает только то множество, которым владеет coordinator, плюс явно
  замороженные через IPC-обработчик, — мы не случайно возобновляем
  что-то, что будущая функция намеренно оставила остановленным.
* **Плюс:** Политика настраиваема (allowlist bundle ID) без изменения
  actor-ов, выполняющих реальную работу.
* **Минус:** Ещё один слой для понимания при чтении кода демона. Смягчаем
  за счёт небольшой поверхности API: `loadModel`, `unloadModel`,
  `emergencyThaw`, `generate` (прокси).

## Рассмотренные альтернативы

* **Инъекция замыкания** в `MLXActor` (`onBeforeLoad: () async -> Void`).
  Легче, чем coordinator, но размазывает связку по коду инициализации демона
  и усложняет отслеживание путей ошибок.
