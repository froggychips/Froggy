# ADR 0005 — Дополнение prompt выполняется на стороне демона, а не клиента

* **Статус:** Принято (Phase 7)
* **Дата:** 2026-05-06

## Контекст

Phase 7 добавила «context-aware generation» — флаг IPC-команды `generate`,
который добавляет недавний OCR-контекст к пользовательскому prompt перед
отправкой в MLX-модель. Работу можно было разместить в двух местах:

1. **В каждом клиенте.** MenuBar / CLI / сторонний скрипт сначала вызывает
   `client.context()`, затем `client.generate(prompt:)` с контекстом,
   склеенным с их prompt.
2. **В демоне.** Клиент отправляет `useContext: true`; демон получает
   `ContextStore.recentContext()` и прогоняет его через `PromptAugmenter`
   перед вызовом `MLXActor`.

## Решение

Вариант 2. Флаг `IPCRequest.useContext: Bool?` — это вся клиентская
поверхность API. `DaemonIPCHandler.augmentedPrompt` выполняет обёртку
один раз, на том же actor-е, который владеет `ContextStore`.

## Последствия

* **Плюс:** Каждый клиент получает дополнение контекста бесплатно. CLI —
  через `froggy gen --context`, MenuBar — через существующую панель prompt
  (просто переключить флаг), Python-скрипт — через
  `{"cmd":"generate", "useContext":true}`.
* **Плюс:** Нет гонки между `client.context()` и `client.generate()`.
  Контекст, который видит модель, — это снапшот на момент генерации, а не
  то, что было актуально 50 мс назад, когда клиент делал первый вызов.
* **Плюс:** Шаблон дополнения централизован — его улучшение улучшает
  всех потребителей. Сегодня он захардкожен в `PromptAugmenter.defaultTemplate`;
  в следующей фазе можно вынести его в `FroggyConfig`.
* **Минус:** Клиенты не могут легко увидеть, что в итоге было отправлено
  модели. Можно решить необязательным debug-флагом, возвращающим полный
  дополненный prompt в ответе по запросу. Пока: не нужно.
* **Минус:** Демон теперь содержит небольшой кусок политики «шаблона
  prompt». Это ведёт к тому, что больше логики LLM-оркестрации живёт
  на стороне демона (system prompt-ы, схемы инструментов и т.д.). Мы
  принимаем это — именно так должен работать демон «AI-оркестратора».

## Рассмотренные альтернативы

* **Двухшаговый API для явных клиентов, дополнение по умолчанию.**
  Отвергнут: удваивает поверхность API без реального прироста возможностей.
* **Дополнение внутри `MLXActor.generate(prompt:)`.** Отвергнут: `MLXActor`
  не знает и не должен знать о `ContextStore` — именно такую связку был
  создан `VortexCoordinator` (ADR 0004), чтобы избежать. IPC-обработчик
  является правильным слоем соединения.
