# ADR-0016: Cloud-роутинг через froggy-mcp — потоки данных и границы приватности

| Поле | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-05-10 |
| Ссылка | THESIS § «Privacy is non-negotiable» |

## Контекст

`froggy-mcp` — companion MCP-сервер, связывающий Claude Code (облачная модель) с демоном
Froggy через существующий Unix-socket IPC. THESIS говорит: *«Cloud routing, when added,
is per-tier and audited.»* Этот ADR и есть тот аудит.

`Redactor` демона работает до записи снапшота в `ContextStore`. Вопрос: что именно
проходит через froggy-mcp в облачную модель, и что Redactor гарантирует, а что нет.

## Что покидает машину при активном froggy-mcp

froggy-mcp предоставляет Claude Code четыре инструмента. Каждый вызов передаёт следующие
данные на серверы Anthropic:

| Инструмент | Что уходит с Mac |
|---|---|
| `froggy_context` | Redacted OCR-текст последних N снапшотов (до `contextMaxChars`). Это контент экрана — имена файлов, код в редакторе, вывод терминала, тела Jira-тикетов, Slack-сообщения, текст браузера — с вырезанными credentials. |
| `froggy_generate` | Строка `prompt`, переданная облачной моделью, плюс текст ответа локальной модели. Ответ может содержать информацию, извлечённую из OCR-контекста. |
| `froggy_transcripts` | Сырой текст транскрипта текущей или последней аудио-сессии: всё сказанное и распознанное на созвоне. Метки спикеров и временны́е метки включены. |
| `froggy_status` | Статус демона: `modelLoaded`, `modelPath`, `capturing`, `memoryPressure`, количество снапшотов, `listening`. Контента нет. |

Все четыре — **явный opt-in при каждом вызове**: Claude Code должен вызвать инструмент;
froggy-mcp не передаёт данные проактивно.

## Что гарантирует Redactor

До записи снапшота в `ContextStore` (и, соответственно, до того, как он может попасть
в froggy-mcp) `Redactor` вырезает:

- AWS access key и secret key (`AKIA…`, `aws_secret_access_key=…`)
- GitHub personal access token (`ghp_…`, `github_pat_…`)
- Anthropic, OpenAI, Slack API токены (по префиксу)
- JWT (`ey…` в трёхчастной dot-separated форме)
- Значения Bearer-заголовков (`Authorization: Bearer …`)
- Key-value пары `password=`, `api_key=`, `secret=`, `token=`
- Номера кредитных карт, проходящие валидацию Luhn (13–19 цифр)

Все они заменяются на `[REDACTED]` до сохранения. Облачная модель их не видит.

## Что Redactor НЕ гарантирует

Redactor — **фильтр credentials**, не фильтр содержимого. Он не:

- Вырезает бизнес-чувствительный текст (описания тикетов, код, сообщения об ошибках,
  диалог на встрече)
- Редактирует имена, email-адреса или внутренние URL, не совпадающие с паттернами credentials
- Фильтрует транскрипты — распознанная речь проходит без изменений, кроме credentials-паттернов
- Защищает то, что никогда не захватывалось: видеокадры и сырое аудио не хранятся и не передаются

**Практический вывод:** при вызове `froggy_context` облачная модель получает реальный контент
экрана — код из редактора, Jira-тикеты в браузере, Slack-треды — минус credentials. Если этот
контент чувствителен сверх credentials (проприетарный код под NDA, данные пациентов, внутренняя
финансовая информация) — вы маршрутизируете его в облачную модель. Риск тот же, что при вставке
скриншота в чат, с автоматической первичной зачисткой credentials через Redactor.

## Модель opt-in

Три гейта должны быть открыты, чтобы данные попали в облако:

1. **froggy-mcp зарегистрирован в настройках Claude Code** — действие пользователя, не дефолт.
2. **Активная сессия Claude Code** — данные не стримятся; они запрашиваются через tool calls.
3. **Вызов инструмента** — облачная модель должна запросить; froggy-mcp не пушит.

Froggy не знает и не хранит какие запросы делает Claude Code. Никакой телеметрии от демона
в froggy-mcp — только ответ на каждую IPC-команду.

## Аудио-транскрипты: повышенная чувствительность

`froggy_transcripts` несёт речь с созвона. Речевой контент категориально чувствительнее
OCR-экрана: люди говорят на встречах то, что не напечатают. Redactor работает на тексте
транскрипта с теми же credential-паттернами, но произнесённые credentials редки — реальный
риск в том, что бизнес-конфиденциальное обсуждение уходит в облако.

Митигация: `froggy_transcripts` — отдельный инструмент от `froggy_context`. Пользователь
может зарегистрировать froggy-mcp только для экранного контекста и явно не использовать
`froggy_transcripts`. Сегодня в демоне нет per-tool ACL — это известный gap.

## Связь с THESIS

THESIS: *«Screen content does not leave the machine without explicit per-source opt-in.
Cloud routing, when added, is per-tier and audited.»*

- **Explicit opt-in:** выполнено — регистрация froggy-mcp + per-call вызов инструмента.
- **Per-tier:** частично выполнено — все четыре инструмента под одним opt-in (регистрация
  froggy-mcp). Отдельный enable/disable per-tool — зафиксированный gap.
- **Audited:** этот ADR и есть запись аудита.

## Решение

Принять текущий data-flow как задокументированный. Формулировка «sensitive data never leaves
the Mac» из README **неточна** и заменена на: credentials вырезаются до того, как данные
покидают машину; контент экрана и транскриптов уходит в облачную модель при вызове
froggy-mcp инструментов, по явному opt-in пользователя.

## Последствия

- README обновлён с точной формулировкой приватности (ссылка на этот ADR).
- Per-tool ACL (отдельное включение/выключение инструментов froggy-mcp) зафиксирован
  как известный gap в TODO.md, не блокер для текущего использования.
- Любой будущий cloud-роутинг (например, Jira-контекст через Atlassian cloud) должен
  получить собственный ADR до merge.
