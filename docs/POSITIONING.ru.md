# Что такое Froggy и чем он не является

Froggy — самостоятельный личный проект с чёткими взглядами. Этот документ существует, чтобы посетители и потенциальные пользователи могли быстро понять, актуален ли он для них — и чтобы участники не открывали issues с запросами, явно выходящими за рамки проекта.

> POSITIONING — о **скоупе** (что входит, что нет). Центральный аргумент
> проекта — почему агрессивная оркестрация памяти плюс уровень trust governance
> и есть вся суть, а не случайный выбор — см. [`THESIS.md`](THESIS.md).

## Чем является Froggy

- **Исследовательский scaffold** для запуска локальных MLX-моделей на
  Apple Silicon Mac с ограниченной памятью — конкретно нацеленный на **8 GB
  unified memory**, конфигурацию, которую большинство существующих инструментов
  для локальных LLM игнорируют.
- **Рабочий пример нативного управления ресурсами macOS**:
  захват через ScreenCaptureKit, Vision OCR, реактивная обработка memory pressure,
  `SIGSTOP` + принудительный pageout для фоновых приложений, изоляция MLX-инференса
  в дочернем процессе, redaction секретов до записи на диск — всё на Swift 6
  со strict concurrency.
- **Хост для плагинов** (`LushaAccessor`): другие инструменты могут читать
  нормализованный контекст экрана/системы через Unix-socket JSON IPC, вызываемый
  из любого языка.
- **Читаемая кодовая база** для изучающих Swift 6, интеграцию MLX,
  ScreenCaptureKit, низкоуровневые macOS API (mach, jetsam, dispatch pressure sources)
  и ADR-driven design.
- **Open source под [лицензией MIT](../LICENSE)** — свободное использование,
  модификация, форк и коммерческое применение с указанием авторства.

## Чем Froggy НЕ является

- **Не потребительский продукт.** Нет установщика, нет авто-обновлений, нет
  канала поддержки за пределами GitHub Issues и Telegram.
- **Не альтернатива Rewind / Granola / Pi.** Это полированные, финансируемые
  продукты из смежных категорий. Froggy не конкурирует с ними и не будет.
- **Не кроссплатформенный.** Только macOS 14+ на Apple Silicon. Intel Mac,
  iOS, Linux, Windows — всё вне скоупа по дизайну: вся логика памяти
  специфична для unified memory.
- **Не заморожен и не имеет стабильного API.** Roadmap исследовательский и может
  меняться. Не полагайтесь на Froggy для критических рабочих процессов. Форматы
  IPC-команд могут изменяться между релизами.
- **Ещё не защищён от вредоносного ввода.** Модель угроз предполагает, что
  локальный пользователь не является злоумышленником; не открывайте IPC-сокет
  или демон недоверенным сетям или недоверенным локальным пользователям.

## Цели (в порядке приоритета)

1. **Запускать полезный локальный LLM-воркфлоу в 8 GB unified memory** без
   постоянных OOM и swap thrash.
2. **Оставаться полностью on-device по умолчанию.** Ничего не покидает машину,
   если пользователь явно не разрешил это. Секреты redact-ируются до записи на
   диск, а не только до отображения.
3. **Быть читаемой референсной кодовой базой** для Swift 6 + MLX + низкоуровневых
   macOS API. Архитектурные решения задокументированы в [`docs/adr/`](adr/).
4. **Оставаться расширяемым.** Plugin API и JSON-line IPC позволяют строить
   поверх Froggy без его форка.

## Не-цели

- Стать SaaS или платным продуктом.
- Обогнать Rewind по памяти прошлой активности или Cursor / ChatGPT по помощи
  в кодинге. Разные категории, разные бюджеты.
- Поддерживать платформы не на Apple Silicon.
- Сохранять обратную совместимость вечно — до версии 1.0 ломающие изменения
  допускаются с заметкой в соответствующем PR.

## Для кого это

Примерно в порядке убывания соответствия:

- Люди с **Apple Silicon Mac на 8 GB**, которые хотят запускать небольшие
  локальные LLM без того, чтобы машина вставала колом.
- **Разработчики, озабоченные приватностью**, которые не могут (или не хотят)
  отправлять содержимое экрана в облачные API — корпоративный код под NDA,
  юридические/медицинские контексты, исследования безопасности.
- **Swift / macOS разработчики**, ищущие реальный пример Swift 6 strict
  concurrency, интеграции MLX, ScreenCaptureKit или низкоуровневого управления
  памятью.
- **Энтузиасты**, которым нужен скриптуемый AI-ассистент, которым можно управлять
  из shell-скриптов, git hooks или своих инструментов через Unix-socket IPC.

## Для кого это НЕ подходит

- Тем, кто ищет полированный продукт с поддержкой. Используйте Rewind, Pi
  или ChatGPT desktop.
- Пользователям Intel Mac или платформ не от Apple.
- Production-развёртывания. Относитесь к Froggy как к личному инструменту,
  а не инфраструктуре.
