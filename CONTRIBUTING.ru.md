# Участие в разработке Froggy

Спасибо за интерес к проекту. Прежде чем открывать issue или PR, просмотрите
[`docs/POSITIONING.md`](docs/POSITIONING.md) и
[`docs/THESIS.md`](docs/THESIS.md) — там объясняется, чем этот проект стремится
быть и чем не является; это определяет, вписывается ли ваша идея в его рамки.

Froggy — личный исследовательский проект под лицензией MIT. Вклад приветствуется,
но оценивается через призму thesis, а не общего «сделать лучше».

## Перед открытием issue

- **Баг-репорты:** укажите версию macOS, модель Apple Silicon, объём RAM
  и вывод `froggy decisions --limit 50`, если релевантно. Трейс-данные
  хранятся локально и не содержат содержимого экрана; редактируйте bundle id,
  которые не хотите публиковать.
- **Запросы функций:** объясните, почему функция проходит тест
  «qualitative vs. quantitative» из `THESIS.md`. «Было бы здорово, если бы
  Froggy работал на Intel Mac» — вне скоупа по дизайну; не открывайте такие
  issue. «Было бы здорово, если бы Froggy умел определять сессии screen-share,
  чтобы не фризить приложение для демонстрации экрана» — в скоупе и
  приветствуется.
- **Вопросы:** Telegram [@froggychips](https://t.me/froggychips) быстрее,
  чем GitHub Issues, для открытых вопросов.

## Перед открытием PR

1. **Сначала обсудите** всё, что выходит за рамки опечатки, мелкого
   баг-фикса или очевидного улучшения качества кода. Откройте issue или
   напишите в Telegram до написания патча — это сэкономит время, если
   изменение не вписывается в thesis.
2. **Архитектурные изменения проходят через ADR.** Если PR добавляет
   подсистему или меняет ключевое решение, напишите ADR в `docs/adr/`
   в рамках этого PR. Формат — по аналогии с существующими ADR.
3. **Новые компоненты сначала получают design-doc.** Если PR вводит
   новый actor, IPC-команду или крупный компонент, напишите design-doc
   в `docs/design/` и смержите его до PR с реализацией. Формат — по
   аналогии с `docs/design/activity-detection.md`.
4. **Соблюдайте порядок документация/реализация.** Согласно
   [ADR 0014](docs/adr/0014-design-docs-after-implementation.md),
   design-doc'ы для слоёв выше текущего по умолчанию отклоняются —
   они создают «гравитационную ловушку документации».

## Соглашения по коду

- Swift 6, strict concurrency, `ExistentialAny`. Никаких послаблений,
  включая тесты.
- Новые системные вызовы (`task_for_pid`, `mach_vm_*`, `dispatch_source_*`,
  `posix_spawn` и т.д.) оборачиваются тонким Swift-слоем для
  тестируемости — смотрите существующие паттерны в `Pageout.swift`,
  `MemoryPressureMonitor.swift`.
- Логирование через `os_log` / `os_signpost`, не через `print`.
- Никаких новых runtime-зависимостей в `Package.swift`, если задача
  физически неразрешима без них. SQLite — через C-API `sqlite3`,
  не через `SQLite.swift`.
- Комментарии объясняют *зачем*, а не *что*. Имена функций объясняют *что*.
  Смотрите операционные принципы в [`docs/THESIS.md`](docs/THESIS.md).

## Тесты

- Запустите `swift test --parallel` локально перед пушем. CI запустит его
  снова, но зелёный локальный прогон экономит один round-trip.
- Новые компоненты получают unit-тесты с инжектируемыми фейками для
  зависимостей от ОС (audio HAL, AX, MLX и т.д.). Паттерн — по аналогии
  с `MemoryPressureMonitorTests`, `PageoutChainTests`.
- Интеграционные тесты, требующие реальных ресурсов macOS, скрываются за
  env-флагом (`FROGGY_RUN_INTEGRATION_TESTS=1`) и пропускаются в CI.

## Сообщения коммитов

Английский или русский — кодовая база проекта двуязычна. Придерживайтесь
стиля последних коммитов из `git log`. Conventional Commits (`feat:`, `fix:`,
`docs:`, `chore:`) приветствуются, но не обязательны. Многострочные
сообщения поощряются — аргументация важнее краткости.

## Формат pull request

Ориентируйтесь на последние смердженные PR (#9, #10, #11, #16, #18, #19):

- Заголовок: короткий, в повелительном наклонении.
- Тело: **Зачем** / **Что** / **Тесты** / **Что осталось**, или английский
  эквивалент. Первый абзац объясняет *зачем*.
- Ссылка на соответствующий ADR или design-doc.
- Краткий итог результатов тестов.

## Лицензия

Отправляя PR, вы соглашаетесь с тем, что ваш вклад лицензируется под
[MIT](LICENSE). Не включайте код под несовместимыми лицензиями
(GPL, AGPL, source-available и т.д.) без явного предупреждения.

## Кодекс поведения

Не будьте грубияном. Автор ведёт этот проект ради удовольствия; если
вклад перестаёт приносить удовольствие обеим сторонам — проект проигрывает.
